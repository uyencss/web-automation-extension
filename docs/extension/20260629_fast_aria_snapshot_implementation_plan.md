# Fast ARIA Snapshot Implementation Plan

Date: 2026-06-29

## Goal

Make `getAriaSnapshot` cheaper, safer, and more stable for agent loops by moving
the default observation path from CDP Accessibility to a content-script DOM
snapshot, while keeping the native CDP accessibility tree as a fallback.

## Problems In Current Implementation

- `getAriaSnapshot` always calls `Accessibility.enable` and
  `Accessibility.getFullAXTree`, then formats the full returned tree in the
  background service worker.
- The handler clears the per-tab ref map on every snapshot, so refs are stable
  only until the next `getAriaSnapshot` call.
- The snapshot includes form `value` content without redacting passwords, OTPs,
  card fields, tokens, and other secret-looking inputs.
- The formatted snapshot is depth-filtered but not viewport-filtered or capped
  by output node count, so large pages can produce noisy model context.

## Design

### Observation Modes

`getAriaSnapshot` accepts:

- `mode: "auto" | "fast" | "native"`
- `scope: "auto" | "viewport" | "full"`
- `maxDepth?: number`
- `maxNodes?: number`
- `viewportMargin?: number`
- `frameId?: number`

Default behavior:

- `mode: "auto"`
- `scope: "auto"` which behaves like viewport-first
- `maxDepth: 8`
- `maxNodes: 250`
- `viewportMargin: 32`

`auto` tries the content-script fast path first. If the content script is not
available, returns no useful elements, or throws on a restricted page, the
handler falls back to the existing native CDP path.

### Content-Script Fast Path

Add an isolated-world content script module:

- Traverses the DOM in the target frame.
- Derives a pragmatic ARIA-like role/name/value/state model.
- Filters to visible elements by default.
- Filters to viewport-intersecting elements for `scope: "viewport"` and
  `scope: "auto"`.
- Preserves useful ancestors when a descendant is included.
- Applies a hard `maxNodes` cap and returns `truncated: true` when exceeded.
- Redacts sensitive form values before returning data to the background worker.

### Persistent Refs

Inside the content script:

- `WeakMap<Element, string>` stores `element -> ref`.
- `Map<string, WeakRef<Element>>` stores `ref -> element`.
- `Map<string, metadata>` stores role/name/bounds/lastSeen.
- Refs are reused while the document/frame lives.
- Stale refs are detected when the `WeakRef` no longer resolves or the element
  is detached.
- Refs are pruned by TTL and max-store size.

Refs generated by the fast path use `R<number>`. Existing native CDP refs keep
the current `S<number>` format.

### Ref Actions

`clickByRef`, `typeByRef`, `hoverByRef`, and `selectByRef` should support both
ref families:

- Fast refs are resolved in the content script and acted on with DOM APIs plus
  actionability checks.
- Native refs keep the current CDP backend-node path.
- If a fast ref is stale, the action returns a stale-ref error and the caller
  should re-run `getAriaSnapshot`.

### Safety

The content script redacts values for:

- `type=password`
- `autocomplete` containing password, one-time-code, OTP, token, card, CVV/CVC,
  or other credential/payment hints
- input `name`, `id`, `aria-label`, or `placeholder` containing password,
  passcode, secret, token, OTP, CVV/CVC, card number, SSN-like, or API key hints

The native CDP fallback should also redact values before formatting.

### Fallback Strategy

Use native CDP when:

- `mode: "native"`
- content script is absent or cannot be messaged
- the page is a restricted Chrome/internal URL
- the fast snapshot returns zero useful lines in `mode: "auto"`

The response includes `source: "content-script"` or `source: "cdp"` so callers
can understand which path handled the request.

## Tasks

1. Add `content-scripts/aria-snapshot.js`.
2. Register it before `bridge.js` in the manifest.
3. Add runtime message handling in `bridge.js` for fast snapshot/ref actions.
4. Add background helpers to call content-script messages with frame targeting.
5. Update `getAriaSnapshot` to support `mode`, `scope`, `maxNodes`, and fallback.
6. Update ref-based action handlers to route fast refs to content script.
7. Redact sensitive values in both fast and native snapshot paths.
8. Update command catalog/documentation for new params and behavior.
9. Verify syntax/imports and run available catalog/build checks.
